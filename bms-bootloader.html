BMS Bootloader

C++20 / C · STM32H7·CubeMX · CAN FD · FreeRTOS·UBC Formula Electric · 2026

A CAN-based firmware-update bootloader for the STM32H7 at the heart of UBC Formula Electric’s BMS. It lives in its own flash partition ahead of the application, validates the installed app with a CRC32 before every boot, and — on request — reflashes the app region sector-by-sector straight over the car’s CAN bus. No debugger, no opening the accumulator: a new build is pushed to every node from the team’s tooling, and a corrupted or half-written update can never brick a board.

How it works

The flash is split into two regions by the linker scripts: a small, protected bootloader partition at the start of memory, and the much larger application partition after it. The very first kilobyte of the app region is a metadata block holding the app’s CRC32 checksum and size, written when the binary is built. The same bootloader library is shared across every board on the car; each board target just supplies its own CAN node ID, FDCAN handle, and CubeMX project.

  • Verified boot. On reset the bootloader runs first. It re-computes a CRC32 over the entire app image and compares it against the checksum in the metadata block. Only if the app is present and the checksum matches — and nothing has explicitly requested staying in the bootloader — does it hand control over.
  • Jump-to-app. Handing off is a bare-metal context switch: disable interrupts and SysTick, clear pending IRQs, point the vector table (VTOR) at the app, switch back to the main stack pointer, then load the app’s initial stack pointer and reset-handler address straight out of its vector table and branch in. The app comes up as if it had booted on its own.
  • CAN update protocol. A compact set of CAN command IDs — START_UPDATE, ERASE_SECTOR, PROGRAM, VERIFY, GO_TO_APP, GO_TO_BOOT — drives a state machine. The host kicks off an update, the bootloader erases the app sectors, streams the new image in block by block, and ACKs progress so the host can show a live bar and retry stalls.
  • Block-buffered flash writes. The H7 can only program a full flash word (32 B) at a time, but a CAN frame carries at most 8 B. Incoming program words are accumulated in a RAM buffer keyed by address; once a whole flash word is filled it is committed to flash, and out-of-order or unaligned writes are rejected.
  • Brick resistance. If an erase or program step fails, or the freshly written image fails its post-flash VERIFY checksum, the app is simply marked invalid — so the next reset lands safely back in the bootloader instead of jumping into corrupt code. A board can always be recovered over CAN.
  • Host-side flasher. A python-can tool (canup) commands one or many boards at once: it puts every node into bootloader mode, parses the Intel-HEX build, and walks each through erase → program → verify → jump, so an entire car can be updated from one command.

What I did

I worked on bringing the CAN bootloader up on the BMS’s STM32H7. That meant standing up the CubeMX boot project, getting the flash partitioning right so the bootloader and application live in separate, non-overlapping regions, and wiring the board-specific layer — FDCAN setup, the FreeRTOS tasks that service the interface, tick, and CAN-TX work, and the board’s CAN node ID — into the shared bootloader library.

The parts that took the most care were the bits where a mistake means a dead board: the jump-to-app handoff (getting the vector table, stack pointer, and interrupt state all correct so the app starts cleanly), and making the update path genuinely brick-resistant by leaning on the CRC32 verified-boot check so a failed or interrupted flash always falls back to the bootloader rather than into garbage. Getting a full image to flash over CAN and the board to come back up running the new firmware — with no debugger anywhere near it — was the satisfying payoff.
